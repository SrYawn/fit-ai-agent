# 方案设计

## 问题
- 当前 `/execute` 直接返回纯字符串，前端如果需要状态信息只能自行解析文本。
- 当前 `/execute/stream` 虽然使用 SSE，但事件粒度是工作流节点，不是模型 token。
- 最终回答正文集中在单个 `result` 事件中返回，无法满足边生成边渲染的流式体验。

## 拟定方案
- 将同步接口返回改为结构化对象，统一暴露 `status`、`intent`、`node`、`profileStatus`、`result`、`message` 等字段。
- 将 SSE 事件模型收敛为四类：
  - `metadata`：开始、路由、画像准备完成等非正文信息
  - `token`：真实增量文本块
  - `done`：流式输出结束
  - `error`：异常结束
- 在 payload 中固定携带 `event`、`node`、`status`、`sequence`、`done`，并按需附带 `intent`、`profileStatus`、`content`。
- 对最终输出节点采用“两阶段执行”：
  - 前半段同步执行工具调用与工作流准备步骤
  - 最后一段回答通过 `ChatClient.stream().content()` 输出真实增量文本
- 计划生成流式场景中，用户画像仍同步准备；真正流式输出只发生在最终 `plan_generation` 节点。

## 备选方案
- 继续沿用现有 `workflow/node/result` 事件，只把完整正文拆成几个字符串块：
  - 放弃，因为这仍然是伪流式，前端依然绑定旧事件语义。
- 直接让整个 ReAct 循环全部改成流式：
  - 暂不采用，因为当前工具调用代理是同步 `call()` + 手动工具执行模型，整体改造成本更高，且本次目标是最终用户可见文本流。

## 风险
- 最终正文改为“工具调用准备完成后重新发起一次流式回答”，因此与同步模式下最后一次非流式回答可能存在轻微措辞差异。
- 计划生成场景下，用户画像节点仍是同步准备，前端在收到正文 token 前可能先收到少量元数据事件。
- 既有前端若强依赖旧的 `workflow/node/result` 事件名，需要同步适配。

## 发布或上线注意事项
- 需要验证同步接口和流式接口两种返回结构都符合预期。
- 需要重点确认 `done` 事件是否只在 token 输出结束后发送。
