# 需求说明

## 摘要
围绕以下四个重点优化当前工作流：
- 修复用户画像的真实数据落地问题
- 对齐画像工具契约与 graph 输入
- 为计划生成增加显式降级控制
- 将伪流式输出改为节点级工作流进度输出

## 背景
此前的工作流在 MySQL 画像数据读取失败后，仍可能继续输出看似个性化的计划。与此同时，用户画像 Agent 的提示词依赖按用户名查询的工具，但 graph 实际传入的是 `userId`，导致链路契约不一致。

## 范围
- 为 DB MCP 增加按 `userId` 查询用户画像的工具
- 去掉画像 Agent 中基于用户名的猜测式查询
- 为下游计划生成加入明确的降级状态
- 为 SSE 接口输出结构化中间进度事件
- 在开发文档中补充 MySQL 启动前置条件

## 非范围
- graph 单例缓存优化
- 动作指导检索策略优化
- 图片质量控制与去重逻辑

## 验收标准
- 用户画像链路能够按 `userId` 查询基本信息
- 数据库不可用时，计划生成明确降级，不伪装成真实个性化结果
- `/execute/stream` 能返回中间工作流进度
- 启动文档清楚说明 DB MCP 工具依赖 MySQL 先启动
