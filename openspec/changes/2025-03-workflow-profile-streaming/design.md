# 方案设计

## 问题
当前工作流同时存在三个核心问题：
- 用户画像 Agent 的提示词与数据库工具输入不一致
- 画像失败后没有显式向下游传播降级状态
- 流式接口几乎不返回中间过程信息

## 拟定方案
- 在 `fitness-db-mcp-server` 中新增 `getUserProfileById`。
- 在 `UserProfileAgent` 中强制优先使用真实数据库工具，禁止猜测用户名。
- 要求画像输出显式携带 `DATA_STATUS: GROUNDED` 或 `DATA_STATUS: DEGRADED`。
- 在 `PlanGenerationAgent` 中基于该状态决定生成个性化计划还是通用起步计划。
- 为工作流输出结构化 SSE 事件，覆盖开始、节点进度、意图识别、最终结果和错误。
- 更新启动文档，明确 MySQL 是 DB MCP 成功工作的前置条件。

## 备选方案
- 保留按用户名查询，并增加更多重试：
  - 放弃，因为这会继续保留猜测式行为和不必要的延迟
- 画像失败时直接阻断计划生成：
  - 放弃，因为对当前用户体验而言，安全的降级结果比直接失败更合理

## 风险
- Agent 最终是否严格遵循 prompt，仍然受模型行为影响。
- SSE 消费端如果之前只接收纯文本，需要适配新的结构化事件。

## 发布或上线注意事项
- 需要分别验证主项目和 DB MCP 模块可以成功编译。
- 后续如启动开销成为问题，可以再推进 graph 单例缓存优化。
